# 지속 가능한 소프트웨어를 위한 클린 코드 원칙

코드의 수명은 우리가 생각하는 것보다 훨씬 깁니다. 작성하는 데 걸리는 시간보다 읽고 수정하는 데 드는 시간이 10배 이상 많기 때문에, '읽기 쉬운 코드'를 작성하는 것은 개발자의 가장 중요한 덕목 중 하나입니다.

## 1. 의미 있는 이름 짓기 (Meaningful Names)
의도를 분명히 밝혀야 합니다. 주석이 없어도 코드 자체로 무엇을 하는지 이해할 수 있어야 합니다.

```javascript
// 나쁜 예
const d = new Date(); // 경과 시간(일)

// 좋은 예
const elapsedTimeInDays = new Date();
```

---

## 2. 함수는 한 가지만 해야 한다 (Single Responsibility)
하나의 함수는 오직 한 가지 작업만 수행해야 하며, 그 작업을 아주 잘 수행해야 합니다.

- **함수의 크기**: 가능하면 20줄을 넘지 않는 것이 좋습니다.
- **인수의 개수**: 이상적인 인수의 개수는 0개(무인수)입니다. 다음은 1개, 2개이며, 3개 이상은 가능한 피하는 것이 좋습니다.

---

## 3. SOLID 원칙의 핵심 요약

| 원칙 | 전체 이름 | 설명 |
| :--- | :--- | :--- |
| **SRP** | 단일 책임 원칙 | 클래스는 단 하나의 변경 이유를 가져야 한다. |
| **OCP** | 개방-폐쇄 원칙 | 확장에는 열려 있고, 수정에는 닫혀 있어야 한다. |
| **LSP** | 리스코프 치환 원칙 | 하위 타입은 언제나 상위 타입으로 교체될 수 있어야 한다. |
| **ISP** | 인터페이스 분리 원칙 | 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다. |
| **DIP** | 의존역전 원칙 | 구체화가 아니라 추상화에 의존해야 한다. |
